# Рекомендации по продукту и зрелости (сравнение с типичными приложениями)

Ниже — не «обязательно завтра», а ориентиры уровня зрелости у соц/дейтинг‑сервисов и фитнес‑комьюнити (аналоги по смыслу: знакомства + контекст зала/города).

## Уже близко к базовой зрелости

- Веб + API + мобильный клиент под одну логику.
- Аутентификация, профили, медиа, лента, чаты, модерация, админка.
- Операционка: health с проверкой БД, смоук, бэкапы, self-heal, опционально Telegram.

## Что обычно есть у более «взрослых» приложений

### Безопасность и доверие

- **Подтверждение контакта**: e‑mail/SMS/звонок (сейчас часто без настоящего SMS — риск фейков).
- **Ограничение «вечной сессии»**: у вас сознательно долгая сессия; у крупных сервисов чаще короткий access + refresh и принудительный re-auth на чувствительных действиях. Компромисс: оставить долгий вход, но **отдельно защищать смену пароля, e‑mail, выплаты**.
- **CAPTCHA / антибот** на регистрации и логине при всплесках (Cloudflare Turnstile и т.п.).
- **WAF / защита от сканеров** перед nginx (часто Cloudflare).

### Уведомления и удержание

- **Push** (FCM/APNs) для сообщений и матчей — у веба этого нет; у нативного приложения это следующий крупный шаг после стабилизации ядра.
- **Тихие часы**, настройки «не беспокоить».

### Качество и саппорт

- **Сбор ошибок клиента**: Sentry (web + mobile).
- **Аналитика**: хотя бы базовые воронки (регистрация → профиль заполнен → первый лайк → первое сообщение) — без этого сложно понимать, где отваливаются пользователи.
- **Публичная страница статуса** или хотя бы мониторинг uptime (внешний ping на `/health?deep=1`).

### Продуктовые фичи «как у других»

- **Верификация профиля** (фото с жестом, бейдж «проверено») — снижает недоверие.
- **Фильтры ленты** (возраст, цели, расстояние) — если география станет важнее «только зал».
- **Репорты и блокировки** с прозрачным SLA для пользователя («рассмотрим за N часов»).

### Юридическое и магазины приложений

- Актуальные **оферта / политика / согласия** (у вас уже есть заготовки в `docs/legal/` — важно синхронизировать с реальным поведением продукта).
- Для **App Store / Google Play**: возрастной рейтинг, механика жалоб, контент‑модерация — проверки стора.

## Практичный порядок внедрения

1. Стабильный прод: nginx лимиты, бэкапы, алерты, смоук — уже в runbook.
2. Push + Sentry на мобилке — максимальный эффект на «ощущение приложения».
3. CAPTCHA / WAF — если появятся атаки или массовые фейки.
4. Аналитика и верификация — когда пойдёт рост и нужна конверсия.

Этот список можно использовать как дорожную карту; пункты не равны по приоритету — выбирайте по рискам и метрикам.
